Sous le Capot : le Voyage d'un Prompt

Chaque message parcourt 4 étapes avant qu'un seul mot ne soit généré.

①

Tokenisation

Texte → sous-mots → IDs entiers via BPE (GPT) ou SentencePiece. C'est ici que le coût se décide : ~1 token ≈ 4 caractères

②

Embedding

Chaque ID → vecteur haute dimension (ex : 4 096d). Un encodage positionnel (RoPE) est ajouté pour que le modèle connaisse l'ordre des mots.

③

Self-Attention

Chaque token « regarde » tous les autres via matrices Q·K·V (multi-head). Complexité O(n²) — doubler les tokens = 4× le coût.

④

Génération

Softmax → probabilités sur le vocabulaire → 1 token choisi. Ajouté au contexte, le cycle recommence (autorégressif).

↻ Ce cycle se répète pour chaque token généré — le contexte entier est relu à chaque itération

04

Les Problématiques du Contexte

Plus le contexte grandit, plus ces phénomènes dégradent la qualité des réponses.

🎯 Lost in the Middle

L'info placée au milieu du contexte est « oubliée ». Le modèle favorise le début et la fin. (Liu et al., 2023)

🔇 Dilution d'Attention

Plus de tokens = scores d'attention dispersés. Le signal utile est noyé dans le bruit des données non pertinentes.

🔄 Context Rot

Les allers-retours prompt↔réponse accumulent du contexte obsolète qui parasite l'attention sur les tours suivants.

📈 Coût Quadratique

O(n²) en mémoire et calcul pour l'attention. 128K tokens coûte ~16× plus que 32K. Latence et facture explosent.

💡

Conclusion : Réduire les tokens n'est pas qu'une question de coût — c'est améliorer la qualité des réponses en gardant un contexte propre et focalisé.

Quel Outil Choisir ?

  • Des fichiers (noms, extensions) → fd
  • Du texte dans des fichiers → rg
  • Un champ dans un YAML → yq
  • Un champ dans un XML → yq -p xml
  • Un champ dans un JSON → jq
  • Le même champ dans N fichiers → fd + yq/jq/xq
  • Des fichiers + leur contenu → fd + rg
  • Output CLI trop verbeux → snip
