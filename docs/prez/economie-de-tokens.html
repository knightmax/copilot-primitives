02

Le Problème

La fenêtre de contexte d'un LLM est une ressource finie. Chaque token inutile :

📉 Réduit la fenêtre

Moins de place pour le raisonnement et le code pertinent

🔇 Dilue le signal

L'info critique est noyée dans le bruit

🐢 Ralentit les itérations

Plus de données à analyser = réponses plus lentes

💸 Augmente le coût

Les tokens en entrée sont facturés par les providers

Exemple : lire 749 fichiers YAML = ~251 000 tokens pour extraire juste un champ par fichier

03

L'Approche : Extraction Chirurgicale

Au lieu de lire des fichiers entiers, extraire uniquement les champs nécessaires en ligne de commande.

❌ Sans optimisation
# Lire le fichier complet
read_file pom.xml
# → ~3 000 tokens par fichier
# × 42 modules = 126 000 tokens
✅ Extraction chirurgicale
$ fd -g "pom.xml" . -x \
  yq -p xml -oy '.project.version' {}
# → ~500 tokens total
# Réduction : -99.6%
3 propriétés clés Chemins relatifs · Respect du .gitignore · Sortie compacte

Outil de base

fd

Trouver des fichiers par nom, extension ou chemin. Chemins relatifs, filtre .gitignore automatique.

749 fichiers YAML — économie : -72%

# Trouver tous les pom.xml

web/pom.xml
# Par extension + profondeur
$ fd -e yaml . catalog/ --max-depth 2
# Compter les résultats
$ fd -e yaml . catalog/systems | wc -l
749

Outil de base

rg (ripgrep)

Chercher du texte ou des patterns regex dans les fichiers. Ne retourne que les lignes pertinentes.

read_file complet (2.6 Mo)~692k tokens
rg "pattern" (415 matchs)~6k tokens

Fichier JSON 2.6 Mo — économie : -98%

# Chercher une dépendance
$ rg "spring-boot" src/ -l
src/main/java/AppConfig.java
src/test/java/IntTest.java
# Compter les occurrences
$ rg -c "agreement_id" event.json
event.json:415
# Extraire les patterns
$ rg "^FROM" Dockerfile -o
FROM eclipse-temurin:21-jre

Outil de base

yq

Extraction YAML / TOML / JSON. Syntaxe jq, un seul outil pour tous les formats structurés.

Économie par fichier
5 tokens au lieu de 335

Par fichier YAML de catalogue

# Extraire un champ YAML
$ yq '.metadata.name' entity.yaml
sa-1242-osmose
# Lister des clés
$ yq '.services | keys' docker-compose.yml
- api
- postgres
- redis
# Conversion YAML → JSON
$ yq -o json docker-compose.yml

Outil de base

xq (yq -p xml)

Extraction XML chirurgicale. Même syntaxe jq, pas d'installation supplémentaire. Idéal pour les pom.xml.

pom.xml complet (50+ deps)~3 000 tokens
xq '.project.version'~10 tokens

Économie : -99.7%

# Version du projet
$ yq -p xml -oy '.project.version' pom.xml
3.2.1
# Lister les dépendances
$ yq -p xml -oy \
  '.project.dependencies.dependency[].artifactId' pom.xml
spring-boot-starter-web
postgresql
lombok
# Filtrer les deps de test
$ yq -p xml -oy \
  '.project.dependencies.dependency[]
  | select(.scope == "test") | .artifactId' pom.xml
09
Synergie

batch-config-audit

fd + yq/jq/xq — extraire le même champ de N fichiers en une seule commande.

Le concept
Sans : O(n × taille_fichier)
Avec : O(n × taille_champ)
Cas réel Maven
-99%
42 pom.xml → versions en 1 commande
# ArtifactId + version de chaque module Maven
$ fd -g "pom.xml" . -x sh -c '
  aid=$(yq -p xml -oy ".project.artifactId" "$1")
  ver=$(yq -p xml -oy ".project.version" "$1")
  echo "$aid ($ver)"
' _ {}
osmose-api (3.2.1)
osmose-core (3.2.1)
osmose-web (3.2.0)
10
Synergie

structural-search

fd + rg — recherche bi-dimensionnelle : par structure ET par contenu.

Pattern 1 : fd → rg

Trouver les fichiers, chercher dedans

Pattern 2 : rg → fd

Chercher du contenu, explorer la structure

Pattern 3 : parallèle

Les deux angles en simultané

# Trouver les pom.xml utilisant Spring Boot
$ fd -g "pom.xml" . -x rg "spring-boot-starter-web" {}
# Composants sans système mappé
$ fd -g "az-project.yaml" catalog/ -x sh -c '
  rg -q "spec.system" "$1" || echo "UNMAPPED: $1"
' _ {}
Naive : ~150 000 tokens fd + rg : ~1 000 tokens = -99%
11
Couche complémentaire

snip

Filtrer la sortie CLI pour ne garder que le signal utile. Réduit la sortie de mvn, dotnet, npm de 80-95%.

mvn test (brut)~8 000 tokens
snip mvn test~800 tokens

Garde: erreurs, warnings, résumés. Supprime: téléchargements, progrès, bruit.

# Utilisation simple
$ snip mvn test
$ snip dotnet build
$ snip npm test
# Comportement bufferisé :
# → Rien pendant l'exécution
# → Sortie filtrée à la fin
# Sortie vide ≠ erreur
# → Attendre la fin du process
12

Architecture Snip

3 couches, zéro duplication. Les skills se chaînent automatiquement.

snip-core

Vérification/installation de snip. Préparation du répertoire de filtres global.

= Fondation

snip-filters-setup

Installeur partagé de profils YAML. Gère les alias (mvn → mvnd).

= Moteur d'installation

snip-jvm / dotnet / npm

Règles d'usage par famille CLI + filtres YAML spécifiques.

= Politique de domaine
⚡ Point clé : matching exact des commandes

Un filtre mvn ne matche pas mvnd. L'installeur partagé génère les alias automatiquement.

13

Quel outil pour quelle situation ?

  • Fichiers par nom, extension, chemin → fd
  • Texte à l'intérieur des fichiers → rg
  • Champ dans un YAML / TOML → yq
  • Champ dans un XML (pom.xml, .csproj…) → xq (yq -p xml)
  • Même champ dans N fichiers de config → batch-config-audit (fd + yq/jq/xq)
  • Fichiers d'un type contenant un texte → structural-search (fd + rg)
  • Filtrer la sortie de commandes build/test → snip
💡 Les skills se déclenchent automatiquement sur mots-clés dans la requête
