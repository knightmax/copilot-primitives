Le Problème

Les frameworks Java ont parfois des APIs trompeuses. La documentation peut être obsolète, incomplète, ou simplement fausse.

❌ Ce qu'on croit

stream = query.getResultStream()
// "Streaming lazy, curseur DB"

✅ Ce qui se passe vraiment

list = query.getResultList()
// Charge TOUT en mémoire
return list.stream()
Bytecode = vérité absolue Un new ArrayList dans le bytecode ne peut pas mentir

Étape 1

fd — Localiser les JARs

Trouver le bon JAR dans le cache Maven (~/.m2) ou les libs du projet.

Pourquoi fd ?
  • 🔍 Recherche récursive rapide dans ~/.m2
  • 📁 Chemins relatifs = moins de tokens
  • 🎯 Filtre par extension (-e jar)
Terminal
# Trouver une lib spécifique
$ fd "hibernate-core" ~/.m2 -e jar
~/.m2/.../hibernate-core-6.2.2.Final.jar
# Toutes les versions d'une lib
$ fd "blaze-persistence" ~/.m2 -e jar
~/.m2/.../blaze-persistence-core-impl-1.6.17.jar
~/.m2/.../blaze-persistence-integration-hibernate6-base-1.6.17.jar
# Exclure sources et javadoc
$ fd "spring" ~/.m2 -e jar |
  rg -v "source|javadoc"

Étape 3b — Le pouvoir

javap -c — Bytecode

Voir les instructions JVM réelles. C'est ici que la vérité se révèle.

Décompilation
$ javap -c -p -classpath lib.jar \
  com.pkg.Class |
  rg "getResultStream|getResultList"

Opcodes à surveiller

invokevirtualAppel de méthode
invokeinterfaceAppel d'interface
invokespecialConstructeur / super
newInstanciation objet
getfieldAccès à un champ
🚨 new ArrayList dans le bytecode = charge TOUT en mémoire. Pas de streaming réel.

Étape 4

rg — Filtrer le bytecode

Pipe javap -c dans rg pour isoler exactement les appels pertinents.

Terminal
# Tracer un appel spécifique
$ javap -c -classpath lib.jar \
  com.pkg.Class |
  rg "getResultStream|getResultList|scroll"
# Chercher des instanciations mémoire
$ javap -c -classpath lib.jar \
  com.pkg.Class |
  rg "new.*ArrayList|new.*HashSet"
# Toutes les invocations
$ javap -c ... |
  rg "invoke(virtual|interface|special|static)"

Outil bonus

jdeps — Dépendances

Analyser les dépendances entre classes et modules. Utile pour comprendre les chaînes de délégation complexes.

Terminal
# Dépendances d'une classe
$ jdeps --class-path lib.jar \
  -verbose:class -filter:none \
  com.blazebit...HibernateExtendedQuerySupport
# Résumé module-level
$ jdeps -s lib.jar
# Analyse récursive
$ jdeps -R --class-path \
  "lib1.jar:lib2.jar" main.jar
